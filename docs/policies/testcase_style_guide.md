# Testcase Style Guide

## Title 작성 규칙

- 한 개의 title은 한 가지 검증 목적만 표현한다.
- title은 `기능 + 조건 + 기대 동작` 형태로 작성한다.
- 모호한 표현을 피한다.
  - 금지: 정상 동작 확인, 잘 나오는지 확인, 문제없는지 확인
  - 권장: 로그인 실패 시 비밀번호 오류 메시지가 노출되는지 확인

## 테스트 진행 화면 작성 규칙

- 가능한 경우 메뉴 경로를 `>`로 연결한다.
- 예: 기업검진 > 종합검진 목록
- 화면 경로가 기획서에 없으면 기능명 또는 source_section을 사용하고 확인 필요로 남긴다.

## Precondition 작성 규칙

- 테스트 수행 전에 반드시 필요한 계정, 권한, 데이터, 상태를 작성한다.
- 여러 조건은 줄바꿈으로 분리한다.
- 기획서에 없는 조건을 추정해야 하면 `확인 필요`를 포함한다.

## Steps 작성 규칙

- 사용자가 실제 수행하는 순서대로 작성한다.
- 각 step은 관찰 가능한 행동 또는 검증 동작이어야 한다.
- API/어드민/DB 확인이 필요한 경우 별도 step으로 작성한다.
- 한 step에 여러 검증 목적을 섞지 않는다.

## Expected Result 작성 규칙

- 화면 표시, 메시지, 상태값, 저장 데이터, API 응답 등 관찰 가능한 결과로 작성한다.
- `정상 처리된다`, `요구사항에 맞게 동작한다`처럼 추상적인 표현은 피한다.
- 기획서 근거가 부족하면 기대 결과에 단정하지 않고 comment/notes에 확인 필요를 남긴다.

## Excel 작성 기준

- `test-result`, `comment`, `issue ticket`은 QA 실행 중 입력하는 영역이므로 생성 시 비워 둔다.
- `test-result`는 pass, fail, blocked, fixed 중 하나만 선택한다.
- 이슈가 발생하면 comment에 재현 정보와 관찰 결과를 작성하고 issue ticket에 티켓 번호/URL을 작성한다.
