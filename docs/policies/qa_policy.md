# QA Policy

## 필수 검토 관점

- 기획서의 모든 기능 요구사항은 정상 케이스와 예외 케이스를 함께 검토한다.
- 사용자 권한, 로그인 상태, 데이터 상태, 플랫폼별 차이가 있으면 별도 테스트케이스로 분리한다.
- 상태값이 있는 기능은 상태 전이 전/후 화면 표시와 API 응답을 함께 검증한다.
- 외부 연동, 배치, 어드민, API 의존성이 있는 기능은 연동 실패/지연/빈 응답 케이스를 검토한다.
- 기획서에 명확하지 않은 정책은 임의로 기대 결과를 만들지 않고 `확인 필요`로 표시한다.

## 우선순위 기준

- High: 로그인, 권한, 결제, 예약, 신청, 삭제, 개인정보, 장애 시 주요 플로우 차단 기능
- Medium: 일반 조회, 필터, 정렬, 상태 표시, 일반 validation
- Low: 문구, 안내 메시지, 비핵심 UI 표시, 도움말

## 테스트 범위 기준

- Functional: 기능 요구사항이 정상 동작하는지 확인
- Negative: 잘못된 입력, 누락값, 권한 없음, 데이터 없음 확인
- Boundary: 최소/최대값, 날짜/시간 경계, 개수 제한 확인
- State: 상태별 노출/수정 가능 여부와 상태 전이 확인
- Permission: 사용자 역할/권한별 접근 가능 여부 확인
- Integration: API, DB, 외부 시스템, 어드민 연동 확인
- Regression: 기존 주요 플로우가 깨지지 않았는지 확인

## 확인 필요로 남겨야 하는 경우

- 성공/실패 기준이 기획서에 없는 경우
- 오류 메시지 문구가 정의되지 않은 경우
- 권한별 노출/수정 정책이 없는 경우
- 상태값 정의 또는 상태 전이 조건이 없는 경우
- API/DB/어드민 의존 요소가 명시되지 않은 경우
